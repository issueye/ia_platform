# IAVM 0.1.10 Delivery Note

**Version:** `0.1.10`  
**Date:** `2026-04-23`  
**Status:** 已完整交付

## 1. 交付范围

`0.1.10` 聚焦“Retry-After backoff hint”，在 `0.1.9` HTTP status retryable mapping 基础上完成了以下目标：

- retryable error 已支持携带 backoff hint
- runtime 在有 hint 时会优先使用宿主建议的退避时间
- `network.http_fetch` 命中 retryable status 且包含 `Retry-After` 时会把该提示透传给 runtime
- 没有 hint 的场景仍保持现有 backoff/jitter 行为

## 2. 主要交付物

### 2.1 代码

- `api.MarkRetryableAfter(err, backoff)`
- `api.RetryBackoffHint(err)`
- runtime retry backoff hint 消费逻辑
- `Retry-After` 到 retryable hint 的映射

### 2.2 测试

- host API retry hint 回归
- `Retry-After` header 映射回归
- runtime hint 优先级回归
- 全量 `go test ./iacommon/... ./iavm/... ./ialang/...` 通过

## 4. 已知边界

以下内容仍不属于 `0.1.10` 的完整目标范围，应进入后续版本：

- 更复杂的 backoff hint 合并策略
- 默认对所有 HTTP 状态自动消费 Retry-After
- retry 与 circuit breaker / budget policy 的联动

## 5. 版本结论

`0.1.10` 已让 IAVM 的 retry 链路从“知道要重试”推进到“知道何时重试”，把服务端的 `Retry-After` 信号真正接进了 runtime 退避策略。
