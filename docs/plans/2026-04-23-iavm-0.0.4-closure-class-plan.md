# IAVM 0.0.4 Closure, Class & Remaining Fixes Plan

**Goal:** 在 `0.0.3` runtime 语义修复基础上，完成闭包 upvalue 捕获、class/this 绑定、对象属性访问修复和位运算 F64 兼容，使 IAVM 示例矩阵达到 10+，覆盖大部分 ialang 语言特性。

**核心交付物:**
- 完整的闭包 upvalue 捕获（嵌套函数访问外层局部变量）
- 对象属性访问修复（`user.name` 正确返回属性值）
- 位运算 F64→I64 自动 coercion
- 基本 class/this 绑定（构造函数 + 实例方法调用）
- 至少 3 个旧示例从"不可运行"变为"可运行"
- 示例矩阵总数 ≥ 10

**原则:** 优先修复已有 opcode 的 runtime 行为；仅在必要时引入新 opcode；所有修复必须有回归测试。

## 1. 0.0.3 之后的基线与遗留问题

### 1.1 已完成基线

`0.0.3` 已形成以下稳定基础：

- 跨类型值运算（string+number、I64+F64 混合算术和比较）
- F64 数组/字符串索引兼容
- 基本闭包函数引用传递（`lowerFunction` funcMap 传递）
- catch 变量绑定（`OpPushTry` + local slot 绑定）
- 7 个稳定 IAVM 示例进入 smoke matrix

### 1.2 当前主要遗留问题

`0.0.4` 需要优先处理以下语义缺陷：

1. **对象属性访问返回 null**
   - `OpGetProp` 使用 module-level constant index，但 lowering 后常量池重映射导致索引错误
   - 影响：`data.ia` 中 `user.name`、`user["city"]` 全部返回 null
   - 位置：`iavm/pkg/bridge/ialang/compiler_lowering.go` 常量池重映射逻辑

2. **位运算要求 I64，编译器发射 F64**
   - `bitAndValues` 等仅处理 `ValueI64`，但 ialang 编译器将所有数字字面量编译为 `float64`
   - 影响：`bitwise.ia` 中所有位运算返回 null
   - 位置：`iavm/pkg/runtime/interpreter.go` 位运算函数

3. **闭包不捕获 upvalue**
   - `OpClosure` 仅压入 FuncRef，不捕获外层函数的局部变量
   - 影响：`closure.ia` 中 `makeCounter` 返回的闭包无法访问外层 `n`
   - 位置：`iavm/pkg/runtime/interpreter.go:190-191`

4. **class/this 绑定缺失**
   - `OpClass` 被降级为 `OpMakeObject`（丢失方法信息）
   - `OpNew` 被降级为 `OpCall`（无 `this` 绑定）
   - 影响：`class.ia` 完全不可运行
   - 位置：`iavm/pkg/bridge/ialang/compiler_lowering.go:625-631`

5. **finally 块不执行**
   - 异常恢复路径跳过 finally 块
   - 影响：`try_catch.ia` 中 finally 输出缺失
   - 位置：`iavm/pkg/runtime/interpreter.go` 异常恢复逻辑

6. **Host ABI 参数传递不完整**
   - `OpHostCall` 不传递栈上的参数给 host capability
   - 影响：host.call 无法传递实际业务参数
   - 位置：`iavm/pkg/runtime/interpreter.go:344-377`

## 3. 0.0.4 交付路径

### Milestone 1：对象属性访问修复

**目标:** 修复 `OpGetProp`/`OpSetProp` 的常量池索引，使对象属性访问正确工作。

**问题分析:**

当前 `buildModuleConstantPool` 重映射所有函数的常量索引到 module-level 常量池。`OpGetProp`/`OpSetProp` 的 `inst.A` 是属性名的常量索引，但重映射后可能指向错误的常量。

`lowerFunction` 中的 remap 逻辑（`compiler_lowering.go:244-252`）处理了 `OpGetProp`/`OpSetProp`，但 `buildModuleConstantPool`（`compiler_lowering.go:142-174`）再次重映射，可能导致双重映射或映射丢失。

**任务:**

1. 检查 `buildModuleConstantPool` 中 `OpGetProp`/`OpSetProp` 的重映射逻辑
2. 确保属性名常量在 module-level 常量池中正确映射
3. 添加集成测试：`OpMakeObject` + `OpSetProp` + `OpGetProp` 全流程
4. 用 `data.ia` 验证

**验收标准:**
- `let user = {name: "alice"}; user.name` → `"alice"`
- `user["city"]` 正确访问
- `let key = "score"; user[key]` 正确访问
- `data.ia` 通过 IAVM build/verify/run

### Milestone 2：位运算 F64 coercion

**目标:** 位运算函数接受 F64 输入，自动截断为 I64。

**任务:**

1. 修改 `bitAndValues`（`interpreter.go:697-701`）：
   - 如果任一参数为 F64，先 `int64(f64)` 截断
   - 如果任一参数为 I64，直接使用

2. 同样修改 `bitOrValues`、`bitXorValues`、`shlValues`、`shrValues`

3. 添加单元测试：
   - `float64(12) & float64(10)` → `int64(8)`
   - `int64(12) | float64(10)` → `int64(14)`
   - `float64(4) << float64(2)` → `int64(16)`

4. 用 `bitwise.ia` 验证

**验收标准:**
- 所有位运算支持 F64 输入
- `bitwise.ia` 通过 IAVM build/verify/run

### Milestone 3：闭包 upvalue 捕获

**目标:** 实现 upvalue cell 机制，使闭包能读写外层函数的局部变量。

**设计:**

```
Upvalue 结构:
  - Closed: bool     // 是否已关闭（函数返回后）
  - Value: Value     // 当前值
  - LocalIdx: int    // 原始 local slot 索引（open 时）
  - FrameIdx: int    // 所属 frame 索引（open 时）

Frame 增加:
  - Upvalues []*Upvalue  // 当前 frame 的 upvalue 列表

OpClosure 行为:
  - 查找当前帧和外层帧中需要捕获的局部变量
  - 为每个捕获变量创建 Upvalue cell
  - 将 Upvalue 引用存入闭包对象

OpLoadLocal / OpStoreLocal 行为:
  - 如果当前帧有 upvalue 映射到该 local slot，
    通过 upvalue cell 读写（保持引用语义）
```

**任务:**

1. 在 `core` 包中定义 `Upvalue` 类型和 `Closure` 类型
2. 修改 `Frame` 添加 `UpvalueMap` 字段（local slot → upvalue 映射）
3. 修改 `OpClosure`：创建 upvalue cell，捕获当前帧的指定 locals
4. 修改 `OpLoadLocal`/`OpStoreLocal`：检查 upvalue 映射，通过 cell 读写
5. 修改 `lowerFunction`：在 lowering 时分析闭包引用的外部变量，编码到 `OpClosure` 的 B/C 字段
6. 修改 `lowerInstructions`：传递闭包捕获信息
7. 添加单元测试和集成测试
8. 用 `closure.ia` 验证

**验收标准:**
- `makeCounter(0)` 返回的闭包能正确读写外层 `n`
- 多个闭包实例独立（`c1` 和 `c2` 不共享状态）
- `closure.ia` 通过 IAVM build/verify/run

### Milestone 4：基本 class/this 绑定

**目标:** 支持 class 构造函数和实例方法调用。

**设计:**

需要在 IAVM 中引入 class 相关的值类型和 opcode：

```
新增 ValueKind:
  - ValueClass    // 类定义（方法表 + 构造函数）
  - ValueInstance // 实例（字段表 + 类引用）

新增/复用 opcode:
  - OpClass: 从栈上收集方法函数引用，创建 Class 对象
  - OpNew: 调用构造函数，创建 Instance 对象，绑定 this
  - OpGetProp: 已有，用于 instance.x
  - OpSetProp: 已有，用于 this.x = value
```

**任务:**

1. 在 `core` 包中定义 `ClassValue` 和 `InstanceValue` 类型
2. 实现 `OpClass` handler：从栈上收集方法名+函数引用，创建 Class 对象
3. 实现 `OpNew` handler：
   - 创建 Instance 对象
   - 调用 constructor 函数，将 `this` 绑定到实例
   - 压入实例
4. 修改 `OpCall`：当调用 instance.method() 时，自动将 instance 作为 `this` 传递
5. 修改 `lowerInstructions`：正确映射 `OpClass` 和 `OpNew`
6. 修改 `lowerFunction`：处理 class 中的方法函数模板
7. 添加单元测试和集成测试
8. 用 `class.ia`（简化版，不含 async 和继承）验证

**验收标准:**
- `class Counter { constructor(n) { this.n = n; } inc() { this.n = this.n + 1; return this.n; } }`
- `let c = new Counter(2); c.inc()` → `3`
- `c.inc()` → `4`
- 简化版 `class.ia` 通过 IAVM build/verify/run

## 5. 风险与处理策略

| 风险 | 影响 | 处理策略 |
|------|------|----------|
| upvalue 捕获实现复杂度高 | 可能需要拆分 Phase | 先完成读取捕获，写入捕获作为可选交付 |
| class/this 需要新 ValueKind | 改动面大 | 先实现最小可用（构造+方法），继承推到 0.0.5 |

| OpCall 需要区分普通调用和方法调用 | this 绑定时机 | 用 OpGetProp 返回 bound method 或用新 opcode |

## 7. 0.0.4 之后的方向

当 `0.0.4` 完成后，再考虑进入更高一层的平台推进：

1. **class 继承**
   - `extends`/`super` 支持
   - 原型链和方法覆盖

2. **finally 块执行保证**
   - 在异常恢复路径中确保 finally 块执行
   - try-catch-finally 完整语义

3. **Host ABI 完善**
   - `OpHostCall` 传递实际参数
   - 多 capability 并发支持

4. **async/await 在 IAVM 中的支持**
   - Promise 值类型
   - await 挂起和恢复

5. **外部模块系统**
   - 模块解析和加载
   - import/export 完整语义
