# IAVM 0.0.8 Delivery Note

**Version:** `0.0.8`  
**Date:** `2026-04-23`  
**Status:** 已完整交付

## 1. 交付范围

`0.0.8` 聚焦“wait/wakeup 最小平台化”，在 `0.0.7` async/platform 基线之上完成了以下目标：

- VM 具备 `WaitSuspension(ctx)`，可等待 pending suspension 恢复
- Host 增加可选 `Waiter` 能力，不破坏既有 Host 主接口
- `run-iavm` 遇到 `ErrPromisePending` 时可自动 wait/resume，形成可执行闭环
- ABI 文档已明确 `Poll` / `Wait` / `ResumeSuspension` 的职责边界

## 2. 主要交付物

### 2.1 代码

- `runtime.VM.WaitSuspension(ctx)`
- `host/api.Waiter` 可选接口
- `DefaultHost.Wait(...)` 最小兼容实现
- CLI `run-iavm` 自动等待 pending suspension

### 2.2 测试

- runtime `host.poll -> wait -> resume` 单测
- CLI `runIavmUntilSettled()` pending suspension 单测
- 全量 `go test ./iacommon/... ./iavm/... ./ialang/...` 回归通过

### 2.3 文档

- `docs/iavm/abi-capabilities.md`
- `docs/plans/2026-04-23-iavm-0.0.8-wakeup-platform-plan.md`
- 本交付说明


## 4. 已知边界

以下内容仍不属于 `0.0.8` 的完整目标范围，应进入后续版本：

- 多路 suspension 的统一调度器
- 基于真实 OS 事件源的 wait/wakeup 实现
- timeout、fairness、backpressure 的系统化策略

## 5. 版本结论

`0.0.8` 已把 IAVM 从“只能手动 resume 的 pending 模型”推进到“具备最小 wait/wakeup 执行闭环”的状态，后续可以在此基础上继续演进事件循环与调度语义。
