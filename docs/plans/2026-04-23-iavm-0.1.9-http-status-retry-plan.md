# IAVM 0.1.9 HTTP Status Retry Plan

**Goal:** 在 `0.1.8` 已完成 DefaultHost 瞬时网络错误自动打标之后，为 `network.http_fetch` 增加“显式状态码重试信号”，让 HTTP 503/429 这类瞬时响应也能接入现有 retry 链路，同时保持默认兼容。

**当前状态:**
- `0.1.8` 已完整交付，默认宿主会对瞬时 `net.Error` 自动标记 retryable
- `network.http_fetch` 对 HTTP 状态码仍统一按成功响应返回，尚未支持状态码级的 retryable 映射

## 1. 版本目标

`0.1.9` 完成三件事：

1. **Capability-level HTTP status retry signal**
   - 支持 `retry_http_statuses`
   - 支持 `retryHTTPStatuses`
   - 仅显式命中时转成 retryable error

2. **Runtime 闭环验证**
   - `DefaultHost` -> retryable error
   - allowlisted `host.call` -> retry -> 成功响应

3. **测试与文档**
   - 增加 host API 与 runtime 回归
   - 更新 ABI / 计划文档 / delivery note

## 2. Milestones

### Milestone 1：HTTP 状态码映射

**状态:** 已完成。

**已完成内容:**
- `DefaultHost` 已支持 capability 级 HTTP 状态码重试信号
- 命中的状态码会转成 retryable error
- 未配置该列表时，HTTP 状态仍按原始响应返回

### Milestone 2：Runtime 闭环

**状态:** 已完成。

**已完成内容:**
- allowlisted `network.http_fetch` 已可消费该 retryable error
- `503 -> retry -> 200` 的 runtime 闭环回归已补齐
- 默认兼容边界保持不变

### Milestone 3：测试与文档收口

**状态:** 已完成。

**已完成内容:**
- host API 状态码映射回归已补齐
- runtime 集成回归已补齐
- 全量测试已通过

## 4. 非目标范围

- 默认对所有 4xx/5xx 自动重试
- 更复杂的 HTTP retry policy（如按 method 区分）
- retry 与 circuit breaker / budget policy 的联动
