# IAVM 0.0.1 平台程序与示例程序交付计划

**Goal:** 面向 `0.0.1`，将当前 IAVM 从“已有最小闭环能力”收敛为一个可构建、可验证、可检查、可运行的平台程序，并提供一个稳定的示例程序作为端到端交付基线。

**核心交付物:**
- 一个可编译的平台程序：`ialang` CLI，包含 `build-iavm`、`verify-iavm`、`inspect-iavm`、`run-iavm` 平台命令。
- 一个可编译的示例程序：固定选择一份小型 `.ia` 示例，生成 `.iavm` 模块并通过验证与运行。
- 一套 0.0.1 验收命令：覆盖构建、验证、检查、运行和 Go 测试。

**原则:** 不追求完整语言语义，不扩大 host ABI；优先保证平台路径稳定、命令清晰、示例可复现、失败可诊断。

---

## 1. 当前基线

### 1.1 已具备的平台能力

当前仓库已经形成三层结构：

| 模块 | 角色 | 当前状态 |
|---|---|---|
| `iacommon` | 公共 Host ABI / capability 类型 | 已作为 IAVM host 能力基础 |
| `iavm` | VM 核心、模块结构、二进制编解码、验证器、运行时、ialang lowering bridge | 已具备最小平台闭环 |
| `ialang` | 语言编译器、原解释器、CLI 入口、IAVM 平台命令入口 | 已承载平台程序入口 |

### 1.2 已具备的 CLI 闭环

`ialang` CLI 当前应作为 0.0.1 的平台程序载体，保留已有语言命令，同时明确 IAVM 平台命令为发布重点：

| 命令 | 0.0.1 定位 |
|---|---|
| `ialang build-iavm <file.ia> -o <file.iavm>` | 将示例源程序编译为 IAVM 模块 |
| `ialang verify-iavm <file.iavm> [policy flags]` | 对模块做静态验证与策略验证 |
| `ialang inspect-iavm <file.iavm> [--verbose] [--verify] [policy flags]` | 输出模块摘要，支持 dry-run verification |
| `ialang run-iavm <file.iavm> [policy flags]` | 验证后运行 IAVM 模块 |

已存在的 verifier policy flags 应进入 0.0.1 文档与验收范围：

- `--profile default|strict|sandbox`
- `--strict`
- `--max-functions N`
- `--max-constants N`
- `--max-code-size N`
- `--max-locals N`
- `--max-stack N`
- `--allow-capability fs|network`

### 1.3 示例程序现状

仓库已有大量 `.ia` 示例，但 0.0.1 不应把全部示例都纳入稳定承诺。经过本轮 smoke 验证，0.0.1 的 canonical example 固定为 `ialang/examples/iavm_hello.ia`。

| 候选 | 路径 | 当前结论 |
|---|---|---|
| 首选 | `ialang/examples/iavm_hello.ia` | 已通过 build / verify / inspect / run 全流程 smoke |
| 备选 | `ialang/examples/control.ia` | 可运行，但输出包含额外 `null`，不适合作为稳定发布示例 |
| 不采用 | `ialang/examples/hello.ia` | 当前会在 `run-iavm` 阶段触发 `cannot get property from non-object` |

0.0.1 固定 `ialang/examples/iavm_hello.ia` 为默认交付示例，其他示例保持“开发样例”，不承诺 release gate。

### 1.4 当前执行状态（2026-04-23）

本轮已完成以下 0.0.1 基线工作：

- 已支持根级 `--help` / `-h` / `help`，并返回退出码 0。
- 已新增 canonical example：`ialang/examples/iavm_hello.ia`。
- 已新增 CLI 回归测试，覆盖 `iavm_hello.ia -> build-iavm -> verify-iavm -> inspect-iavm -> run-iavm`。

  - `./bin/ialang inspect-iavm ./tmp/iavm_hello.iavm --verify --profile sandbox`

## 2. 0.0.1 目标范围

### 2.1 必须完成

1. **平台程序可构建**
   - 能从仓库源码构建出 `ialang` CLI。
   - 构建产物可执行 IAVM 平台命令。

2. **示例程序可编译**
   - `ialang/examples/iavm_hello.ia` 可通过 `build-iavm` 生成 `.iavm` 文件。
   - 生成文件结构可被 `DecodeModule` 读取。

3. **示例模块可验证**
   - 默认策略下 `verify-iavm` 通过。
   - `inspect-iavm --verify` 能输出模块摘要和 verification summary。
   - 如示例无需 host capability，`sandbox` profile 应作为推荐验证策略。

4. **示例模块可运行**
   - `run-iavm` 在验证通过后执行模块。
   - 示例输出稳定，适合作为 smoke test。

5. **测试基线稳定**
   - `go test ./iavm/...` 通过。
   - `go test ./ialang/...` 通过。
   - 如需要发布前全量检查，再补充 `go test ./iacommon/...`。

### 2.2 非目标范围

以下内容不进入 0.0.1，避免扩大交付面：

- 完整模块系统与包发布机制。
- 完整 closure upvalue 捕获语义。
- class / inheritance 完整运行时语义。
- async / await / promise 完整语义。
- JIT、AOT、本机代码生成。
- 完整标准库或完整 Node-like host API。
- 跨平台安装器、包管理器、远程模块仓库。

## 3. 交付路径

### Milestone 0：冻结 0.0.1 基线

**目标:** 明确 0.0.1 只交付 IAVM 平台最小可用路径。

**任务:**
- 固定 canonical example：`ialang/examples/iavm_hello.ia`。
- 固定平台命令集合：`build-iavm` / `verify-iavm` / `inspect-iavm` / `run-iavm`。
- 明确 verifier policy flags 的用户可见行为。
- 确认当前测试基线无回归。

**验收:**
- 当前计划文档存在并作为 0.0.1 范围说明。
- 所有后续任务只围绕该范围推进。

### Milestone 1：平台程序构建

**目标:** 能从源码构建出一个明确的平台 CLI 程序。

**建议命令:**

```bash
go build -o ./bin/ialang ./ialang/cmd/ialang
```

**任务:**
- 确认 `ialang/cmd/ialang` 可单独构建。
- 如 `bin/` 目录未纳入版本管理，只将构建命令写入文档，不提交二进制产物。
- 运行平台程序帮助输出，确认 IAVM 命令可见。

**验收命令:**

```bash
./bin/ialang --help
```

**验收标准:**
- 命令退出码为 0。
- help 文本包含 `build-iavm`、`verify-iavm`、`inspect-iavm`、`run-iavm`。

### Milestone 2：示例程序编译为 IAVM 模块

**目标:** 将 canonical example 编译为稳定的 `.iavm` 模块。

**建议命令:**

```bash
./bin/ialang build-iavm ./ialang/examples/iavm_hello.ia -o ./tmp/iavm_hello.iavm
```

**任务:**
- 确认 `iavm_hello.ia` 不依赖 0.0.1 非目标语义。
- 确认输出文件可被 IAVM binary decoder 读取。
- 保持示例足够小，作为 0.0.1 release gate 的稳定基线。

**验收标准:**
- 生成 `iavm_hello.iavm`。
- 文件可被 `inspect-iavm` 读取。

### Milestone 3：示例模块验证与检查

**目标:** 让 `.iavm` 模块具有可解释的元信息和明确的验证结果。

**建议命令:**

```bash
./bin/ialang verify-iavm ./tmp/iavm_hello.iavm --profile sandbox
./bin/ialang inspect-iavm ./tmp/iavm_hello.iavm --verify --profile sandbox
./bin/ialang inspect-iavm ./tmp/iavm_hello.iavm --verbose
```

**任务:**
- 确认 `verify-iavm` 成功输出 mode/profile 信息。
- 确认 `inspect-iavm --verify` 不执行模块，只做 dry-run verification。
- 确认 `inspect-iavm --verbose` 能列出函数摘要，便于诊断模块生成结果。

**验收标准:**
- `verify-iavm` 退出码为 0。
- `inspect-iavm --verify` 输出 `verification: passed`。
- `inspect-iavm` 输出包括 magic、version、target、entry、functions、constants 等摘要字段。

### Milestone 4：示例模块运行

**目标:** 让 0.0.1 用户能看到完整平台路径：source -> iavm -> verify -> run。

**建议命令:**

```bash
./bin/ialang run-iavm ./tmp/iavm_hello.iavm --profile sandbox
```

**任务:**
- 确认 `run-iavm` 默认要求 entry point。
- 确认运行前验证失败时不会进入 runtime。
- 确认运行输出可作为 smoke test 判断依据。

**验收标准:**
- 命令退出码为 0。
- 示例输出稳定。
- 失败时错误信息能区分 decode、verify、runtime 三类阶段。

### Milestone 5：发布前测试矩阵

**目标:** 形成 0.0.1 发布前最小质量门禁。

**必跑命令:**

```bash
go test ./iavm/...
go test ./ialang/...
go test ./iacommon/...
```

**可选 smoke 命令:**

```bash
go build -o ./bin/ialang ./ialang/cmd/ialang
./bin/ialang build-iavm ./ialang/examples/iavm_hello.ia -o ./tmp/iavm_hello.iavm
./bin/ialang verify-iavm ./tmp/iavm_hello.iavm --profile sandbox
./bin/ialang inspect-iavm ./tmp/iavm_hello.iavm --verify --profile sandbox
./bin/ialang run-iavm ./tmp/iavm_hello.iavm --profile sandbox
```

**验收标准:**
- 所有 Go 测试通过。
- smoke 命令全部通过。
- 无新增未解释的 panic、数据竞争或 flaky 测试。

## 5. 风险与处理策略

| 风险 | 影响 | 处理策略 |
|---|---|---|
| `hello.ia` 使用了暂不稳定的 host/import 语义 | 旧示例无法作为 release gate | 已切换为 `iavm_hello.ia`，并保留该问题作为后续兼容性修复项 |
| `sandbox` profile 与 capability allowlist 语义不完全一致 | 用户对 sandbox 安全边界产生误解 | 0.0.1 文档明确 sandbox 是资源限制/entry 策略，capability deny-by-default 作为后续修复项 |
| CLI policy flags 分散在命令实现中 | 后续命令行为漂移 | 后续抽象为一等 verifier profile/policy 类型 |
| 示例运行依赖 stdout/print host 行为 | smoke test 不稳定 | 固定示例输出，并补 CLI 层回归测试 |
| 语言语义覆盖不足 | 复杂示例无法编译/运行 | 0.0.1 只承诺 canonical example 与小语义子集 |

## 7. 0.0.1 后续方向

0.0.1 完成后，再进入平台增强阶段：

1. **Verifier policy 一等化**
   - 将 `default` / `strict` / `sandbox` 从 CLI 字符串收敛为共享 policy 类型。
   - 明确 capability allowlist 的空列表语义。

2. **Host ABI 收敛**
   - 明确 `print`、fs、network 等能力的声明、导入、调用与验证关系。
   - 校验 capability declaration 与实际 host call 的一致性。

3. **示例矩阵扩展**
   - 增加 function/control/object/capability 示例。
   - 每个示例标注需要的 profile 与 capability。

4. **发布自动化**
   - 增加统一 smoke test 脚本或 Make target。
   - 在 CI 中覆盖 build -> build-iavm -> verify -> inspect -> run。

5. **语言语义补齐**
   - closure upvalue、class、module、async 等按示例矩阵逐步推进。
