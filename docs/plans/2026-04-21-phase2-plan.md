## 2. 本轮开发目标

**总目标**：实现 Try-Catch 完整的异常处理机制，让 lowering 后的 try-catch 程序能在 iavm runtime 中正确执行。

### Phase 1: Try-Catch 异常恢复机制

#### P0: 实现 Interpret 主循环中的异常恢复
**目标**：当 `dispatch` 返回 error（由 OpThrow 或其他运行时错误产生）时，不再直接退出，而是检查 `tryStack` 并跳转到 catch handler。

**当前问题**：
- `OpThrow` 直接 `return fmt.Errorf("throw: %s", msg)`
- 主循环 `if err := vm.dispatch(inst, frame); err != nil { return err }` 直接向上传播
- `tryStack` 只记录了 handler target 地址，但从未在异常时使用

**设计思路**：
1. 引入 `VMError` 结构体，封装异常值和调用栈信息
2. 修改主循环：dispatch 返回 error 时，检查当前 frame 和上层 frames 的 tryStack
3. 异常恢复流程：
   a. 保存异常值到栈中
   b. 将当前 frame 的 IP 设置为 tryStack 记录的 handler 地址
   c. 弹出 tryStack（异常已处理）
   d. 继续正常执行
4. 如果遍历所有 frames 后仍未找到 handler，则返回原始 error（未捕获异常）

**关键文件**：
- `iavm/pkg/runtime/vm.go` — 新增 VMError 结构
- `iavm/pkg/runtime/interpreter.go` — 修改 Interpret 主循环和 OpThrow
- `iavm/pkg/runtime/interpreter_test.go` — 新增 try-catch 测试

**验收标准**：
- [ ] `try { throw "err"; } catch (e) { }` 能正常捕获异常并继续执行
- [ ] 未捕获的异常仍能正确向上传播
- [ ] 嵌套 try-catch 能正确工作（内层未捕获抛到外层）
- [ ] try-catch 测试覆盖：简单捕获、嵌套捕获、未捕获异常

## 6. 关键实现细节

### 异常恢复机制
- **Frame.TryHandlers**：每个 frame 维护自己的 try handler 栈（`[]uint32`）
- **OpPushTry**：向当前 frame 的 TryHandlers 追加 handler IP
- **OpPopTry**：从当前 frame 的 TryHandlers 弹出
- **OpThrow**：弹出异常值，存入 `vm.exception`，返回 `ErrUncaughtException`
- **主循环恢复**：捕获 `ErrUncaughtException` 后，从当前 frame 向上 unwind：
  1. 检查当前 frame 的 TryHandlers，如有则取最近一个
  2. 设置 frame.IP = handlerIP，将 vm.exception 压栈，清空 exception
  3. 继续正常执行
  4. 如当前 frame 无 handler，pop frame 继续检查上一个
  5. 如所有 frame 都无 handler，返回 `ErrUncaughtException`

### 测试覆盖
- **Interpreter 单元测试（4 个）**：简单捕获、rethrow、嵌套 catch、未捕获异常
- **集成测试（2 个）**：try-catch 端到端、异常值传递
