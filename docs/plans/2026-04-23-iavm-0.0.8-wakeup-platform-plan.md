# IAVM 0.0.8 Wakeup Platform Plan

**Goal:** 在 `0.0.7` 已完成 async/platform 最小闭环之后，把 IAVM 从“手动 resume 的 pending 模型”推进到“具备最小 wait/wakeup 驱动”的执行基线，为后续事件循环与 backpressure 打底。

**当前状态:**
- `0.0.7` 已完整交付，`await` / Promise / suspension / network handle client-server smoke 已稳定
- VM 当前只能通过 `ResumeSuspension()` 主动重试挂起 Promise
- `run-iavm` 对真正 pending 的异步资源还没有自动等待恢复路径

## 1. 版本目标

`0.0.8` 聚焦三件事：

1. **Wait / wakeup 最小执行入口**
   - VM 具备 `WaitSuspension(...)` 能力
   - Host 可选暴露 wait 接口
   - CLI 遇到 pending suspension 时可自动等待恢复

2. **Pending 资源调度语义收敛**
   - 明确 `poll`、`wait`、`resume` 的职责边界
   - 把“重轮询”与“被动等待”区分为独立语义层

3. **后续事件循环前置骨架**
   - 为多 suspension / timeout / fairness 预留扩展位
   - 不在 `0.0.8` 内追求完整事件循环实现

## 2. Milestones

### Milestone 1：VM wait/resume 入口

**状态:** 进行中。

**目标:**
- 增加 `WaitSuspension(ctx)` 或等价能力
- `run-iavm` 自动处理 `ErrPromisePending`
- 为 runtime 增加最小 wait 单测

### Milestone 2：Host wait 能力约定

**状态:** 待开始。

**目标:**
- 在 Host API 中声明可选 wait 接口
- `DefaultHost` 提供最小兼容实现
- 文档明确 `Poll` vs `Wait` 的责任

### Milestone 3：文档与发布收口

**状态:** 待开始。

**目标:**
- 更新 ABI / delivery 文档
- 评估 `0.0.8` 完成度与已知边界

## 3. 发布前验证

当前阶段至少保持：

```bash
go test ./iacommon/... ./iavm/... ./ialang/...
```

## 4. 非目标范围

以下内容不属于 `0.0.8` 完整交付范围：

- 多路挂起统一调度器
- 真实 epoll/kqueue/IOCP 事件循环
- Promise continuation scheduler 与任务公平性策略
