# IAVM 0.0.6 Semantic Expansion Plan

**Goal:** 在 `0.0.5` 已完成 Host ABI 与 control-flow 收敛交付的基础上，进入下一阶段的“语义扩展”工作，优先完成 `extends` / `super`、Host handle table / `OpHostPoll` 最小资源模型，以及更广的 typed adapter / capability 配置边界。

**当前状态:**
- `0.0.5` 已完整交付，Host ABI 参数传递、typed adapter 首轮接入、finally 回归与 capability 一致性校验均已落地
- runtime 已稳定绑定最近一次导入的 capability，并兼容模块级常量池
- IAVM 主路径 `encode/decode/verify/runtime/CLI` 具备全量 Go 测试与 smoke 回归
- `extends` / `super` 已进入稳定范围
- handle-based host resource model 已完成第一条最小闭环：`fs.open/read/write/seek/close` + `OpHostPoll`
- `run-iavm --cap-config` 已支持 TOML capability 配置、FS preopen 与 Network policy 映射

**核心交付物:**
- 基本 class 继承语义：`extends` / `super`
- Host handle table 与 `OpHostPoll` 的最小可运行模型
- 更广的 FS / Network typed adapter 与 capability 配置映射
- 为 `0.0.7` async/platform 扩展建立可持续的资源语义边界

**原则:** `0.0.6` 允许适度扩张语义面，但每一项新能力都必须满足“可验证、可回归、可文档化”三条件；优先做最小稳定闭环，而不是一次性追求完整平台化。

---

## 1. 0.0.5 之后的基线

`0.0.5` 已形成以下稳定基础：

- `OpHostCall` 参数传递契约稳定
- `DefaultHost` 已接入 FS / Network HTTP typed adapter 第一阶段实现
- verifier 已校验静态 host operation 与 capability 声明的一致性
- `try/finally`、`try/catch/finally` 已被 runtime 与 CLI 回归锁定
- README、ABI 文档与 delivery note 已同步到交付态

这意味着 `0.0.6` 可以不再回头修补 `0.0.5` 的主路径，而应开始解决此前明确延期的语义扩展项。

---

## 2. 0.0.6 问题拆分

### 2.1 主要缺口

1. **class 继承已完成最小闭环**
   - `extends` / `super` 已进入 lowering、runtime 与回归矩阵
   - 后续主要是补更多继承场景与示例覆盖

2. **Host handle 资源模型已建立最小基线**
   - `fs.open/read/write/seek/close` 已接入 `DefaultHost`
   - `OpHostPoll` 已具备最小可验证语义
   - network handle 系列仍未进入稳定范围

3. **typed adapter 覆盖面仍偏窄**
   - 当前主要稳定在 `fs.read_file` / `network.http_fetch`
   - 其余 operation 仍依赖原始 `map[string]any`

4. **capability config 到 CLI 的映射尚不完整**
   - verifier 已能校验 capability kind
   - `AcquireRequest.Config` 仍缺少更稳定的 CLI / 文件配置入口

### 2.2 版本定位

`0.0.6` 不再只是“契约收敛”，而是：

- 在语言层补足最明显缺失的类继承能力
- 在平台层为真正的异步 / 资源句柄模型打下最小可运行基线
- 在宿主边界上继续减少 `map[string]any` 的扩散面

---

## 3. 0.0.6 目标范围

### 3.1 必须完成

1. **基本 `extends` / `super`**
   - 子类继承父类方法
   - `super(...)` 调父类构造函数
   - `super.method()` 调父类原型方法

2. **Host handle table 最小闭环**
   - 至少完成一条稳定的 handle-based capability 路径
   - 推荐优先 `fs.open/read/write/close`
   - `OpHostPoll` 不再只是空实现，而要具备最小可验证语义

3. **typed adapter 扩展一轮**
   - 至少把 handle 模型涉及的 capability request/response 继续类型化
   - 明确 raw map 仅作为兼容层，而不是推荐接口

4. **文档与验证同步**
   - README
   - ABI / capability 文档
   - `0.0.6` 计划与交付说明

### 3.2 建议完成

1. **CLI capability 配置入口雏形**
   - 例如 capability config 文件或显式参数映射
   - 能落到 `AcquireRequest.Config`

2. **更清晰的 smoke 分类**
   - class inheritance
   - host handle
   - host poll

### 3.3 非目标范围

以下内容不纳入 `0.0.6` 最小闭环：

- Promise / await 的完整挂起恢复平台化
- socket/listen 全矩阵网络资源模型
- 外部模块系统大规模重构
- JIT / AOT / native backend
- 完整的类特性补齐（如复杂装饰器、元类等）

## 4. 交付路径

### Milestone 1：`extends` / `super` 闭环

**目标:** 让 class 从“可实例化、可绑定 this”扩展到“可继承、可调用父类逻辑”。

**当前状态:** 已完成。

**已完成内容:**
1. lowering 已保留 `OpSuper` / `OpSuperCall`
2. runtime 已支持父类构造调用与父类方法查找
3. runtime / integration / CLI 已增加继承回归

**验收标准:**
- `class Dog extends Animal { ... }` 可运行
- `super(name)` 可调用父类构造
- `super.speak()` 可调用父类方法
- 新增继承示例通过 build / verify / run

### Milestone 2：Host handle table 最小资源模型

**目标:** 让 IAVM 首次拥有“可打开、可持有、可轮询、可关闭”的宿主资源句柄语义。

**当前状态:** 第一条 FS 路径已完成最小闭环。

**已完成内容:**
1. 明确 lowering 阶段如何编码父类引用与 `super` 调用
1. 在 host API 中定义了 handle-based FS operation typed adapter
2. `DefaultHost` 已维护 file handle table
3. `OpHostPoll` 已从栈上消费 handle 并返回 poll 结果对象
4. host API / runtime / integration 已增加资源句柄测试

**验收标准:**
- `fs.open` 返回稳定 handle
- `fs.read` / `fs.write` / `fs.close` 可通过 handle 执行
- `OpHostPoll` 对 handle 状态有最小可验证行为
- 资源释放路径具备测试覆盖

### Milestone 3：typed adapter 与 capability config 扩展

**目标:** 把 `0.0.5` 的 typed adapter 起点继续推进，避免新资源模型重新退回到原始 map 风格。

**当前状态:** 已完成当前版本的最小目标。

**已完成内容:**
1. handle-based FS operation 已通过 typed request/response adapter 接入
2. `run-iavm --cap-config` 已把 TOML capability 配置映射到 `AcquireRequest.Config`
3. 本地 FS preopen 与 Network HTTP policy 已从同一配置文件构建

**验收标准:**
- 新增 operation 优先使用 typed adapter
- capability config 文档中有清晰字段约定
- 文档中说明兼容策略与过渡层边界

### Milestone 4：0.0.6 发布前验证

**目标:** 保证语义扩展不会破坏 `0.0.5` 已交付主路径。

**必跑命令:**

```bash
go test ./iacommon/...
go test ./iavm/...
go test ./ialang/...
```

**关键回归:**

```bash
go test ./iavm/pkg/runtime -run TestCapability_
go test ./iavm/pkg/runtime -run TestClass
go test ./iavm/pkg/integration -run TestFullPipeline_WithHostCapability
go test ./ialang/cmd/ialang -run TestRunCLIIavm
```

**验收标准:**
- 全量 Go 测试通过
- 0.0.5 回归不退化
- 新增继承示例稳定通过
- 新增 host handle 示例稳定通过

## 6. 风险与处理策略

| 风险 | 影响 | 处理策略 |
|------|------|----------|
| `extends` / `super` 涉及 lowering 与 runtime 双侧改动 | 容易引入 class 回归 | 先做最小继承模型，保持 smoke + runtime 双回归 |
| handle table 设计过大 | 版本范围失控 | 优先只做一条最小闭环，如 `fs.open/read/write/close` |
| `OpHostPoll` 语义不清 | 后续 async 基础不稳 | 先定义最小同步可验证语义，再扩展真正异步能力 |
| typed adapter 跟不上新增 operation | Host ABI 再次扩散 | 强制新 operation 默认从 typed adapter 进入 |

## 8. 0.0.6 之后的方向

当 `0.0.6` 语义扩展闭环完成后，再进入更明显的平台深化阶段：

1. **async/await 平台化**
   - Promise 值类型
   - await 挂起与恢复

2. **更完整的 host resource 模型**
   - network.dial/listen/send/recv
   - 更丰富的 poll / wakeup / backpressure 语义

3. **capability config 文件化**
   - `run-iavm --cap-config`
   - 更稳定的 capability profile 管理

4. **更系统的示例矩阵**
   - inheritance
   - host handles
   - async-ready smoke
