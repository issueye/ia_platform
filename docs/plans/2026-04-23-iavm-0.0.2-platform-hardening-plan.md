# IAVM 0.0.2 平台硬化与示例扩展计划

**Goal:** 在 `0.0.1` 已完成最小平台闭环的基础上，推进 `0.0.2` 成为一个更稳定、更可诊断、对开发者更友好的 IAVM 平台版本。重点不是扩大语言表面，而是补齐平台策略语义、提升 CLI/验证器/示例矩阵的一致性，并为后续更复杂程序打下稳定基线。

**核心交付物:**
- 一套更清晰、可复用的 verifier/profile 语义。
- 一个从“单一 canonical example”提升到“小型示例矩阵”的 IAVM smoke 基线。
- 更稳定的 CLI 用户体验与错误诊断输出。
- 对 `0.0.1` 中已知问题的收敛方案，尤其是 capability/profile 语义与旧示例兼容性问题。

**原则:** 优先硬化现有平台路径，不追求完整语言特性覆盖；所有新增能力都必须能落到 CLI、测试和示例验收上。

---

## 1. 0.0.1 之后的基线与遗留问题

### 1.1 已完成基线

`0.0.1` 已形成以下稳定基础：

- `ialang` CLI 可构建为平台程序。
- IAVM 命令闭环已具备：
  - `build-iavm`
  - `verify-iavm`
  - `inspect-iavm`
  - `run-iavm`
- 根级 `--help` / `-h` / `help` 可正常返回 usage。
- canonical example 已固定为 `ialang/examples/iavm_hello.ia`。
- `iavm_hello.ia` 已通过 build / verify / inspect / run 全流程 smoke。
- `go test ./iacommon/...`、`go test ./iavm/...`、`go test ./ialang/...` 已通过。

### 1.2 当前主要遗留问题

`0.0.2` 需要优先处理以下遗留项：

1. **verifier profile 语义仍偏分散**
   - profile 仍主要通过 CLI 字符串驱动。
   - `default` / `strict` / `sandbox` 缺少统一的一等 policy 类型。

2. **sandbox capability 语义存在歧义**
   - 当前文档语义与实现细节之间仍可能让用户误解“empty allowlist 是否表示 deny-all”。
   - 需要明确 capability allowlist 的规则，并让 `inspect` / `verify` 输出能解释该策略。

3. **示例覆盖过窄**
   - `0.0.1` 只有一个稳定示例 `iavm_hello.ia`。
   - `control.ia`、`hello.ia` 等示例的 IAVM 兼容性未收敛，不适合作为 release gate。

4. **错误诊断仍偏底层**
   - 部分失败信息仍以 verifier/runtime 内部错误为主。
   - 缺少更清晰的“编译失败 / 验证失败 / 运行失败 / profile 拒绝”分层输出。

5. **平台基线尚未形成更完整 smoke matrix**
   - 当前 smoke 只覆盖一个最小示例。
   - 还没有针对 capability、控制流、多函数等维度的稳定矩阵。

## 2. 0.0.2 目标范围

### 2.1 必须完成

1. **统一 verifier/profile 语义**
   - 将 CLI profile 语义收敛到共享 policy 构造逻辑。
   - 明确 `default` / `strict` / `sandbox` 的行为边界。
   - 明确 capability allowlist 的空值、未设置、显式 deny-all 之间的区别。

2. **增强 inspect/verify 的策略可见性**
   - `inspect-iavm --verify` 除了输出 passed/failed，还应能显示有效 profile/policy 摘要。
   - `verify-iavm` 在成功输出中明确当前模式、关键限制项和 capability 策略。

3. **建立 0.0.2 小型示例矩阵**
   - 保留 `ialang/examples/iavm_hello.ia` 作为最小 smoke 示例。
   - 额外增加 2~3 个稳定 IAVM 示例，覆盖：
     - 控制流
     - 多函数/调用
     - capability 或 host 约束（若当前语义允许）
   - 每个示例都明确推荐 profile 与期望输出。

4. **修复至少一个旧示例的 IAVM 兼容性问题**
   - 优先考虑 `ialang/examples/hello.ia` 的 `cannot get property from non-object` 问题。
   - 如果无法在 0.0.2 内完全修复，则至少要：
     - 确认根因
     - 固化回归测试
     - 在文档中明确其当前状态与限制

5. **增强 CLI 失败诊断**
   - 规范 `build-iavm` / `verify-iavm` / `inspect-iavm` / `run-iavm` 的错误前缀与分层。
   - 保证用户能直观看到失败发生在 decode、verify、policy、runtime 哪一层。

### 2.2 非目标范围

以下内容仍不进入 `0.0.2`：

- 完整 class / inheritance / async / promise 兼容性收敛。
- 完整模块包发布系统。
- 大规模标准库/宿主 API 扩展。
- JIT / AOT / native backend。
- 面向生产的安全沙箱承诺。

## 3. 0.0.2 交付路径

### Milestone 0：冻结 0.0.2 问题清单

**目标:** 将 `0.0.1` 的已知问题显式收敛为 `0.0.2` 工作项。

**任务:**
- 整理 verifier/profile/capability 语义现状。
- 整理现有示例的 IAVM 可运行性状态。
- 确定本版本要修复的旧示例问题清单。

**验收:**
- 形成清晰的 0.0.2 范围文档。
- 避免在实现过程中再次扩大问题边界。

### Milestone 1：verifier policy 一等化

**目标:** 将 profile/policy 从零散 CLI 逻辑收敛为共享平台语义。

**任务:**
- 引入统一的 IAVM verify policy 构造入口。
- 明确：
  - `default`
  - `strict`
  - `sandbox`
  的基础约束与覆盖规则。
- 让 `verify-iavm`、`inspect-iavm`、`run-iavm` 复用同一 policy 解释路径。

**验收标准:**
- CLI 三个命令不再各自隐含不同 profile 语义。
- 新增测试覆盖 profile 默认值、显式覆盖、capability allowlist 组合行为。

### Milestone 2：策略输出与诊断增强

**目标:** 让命令输出更适合作为平台工具使用，而不只是内部调试接口。

**任务:**
- 增强 `verify-iavm` 成功输出，包含关键 policy 信息。
- 增强 `inspect-iavm --verify` 输出，显示 mode、限制项和 capability 视图。
- 统一错误文案前缀。

**验收标准:**
- 用户可以只看 CLI 输出判断当前模块为何通过/失败。
- 测试覆盖成功和失败两类输出格式。

### Milestone 3：示例矩阵扩展

**目标:** 从单一 smoke example 提升到小型稳定示例矩阵。

**建议示例集合:**

| 类型 | 候选路径 | 0.0.2 目标 |
|---|---|---|
| 最小示例 | `ialang/examples/iavm_hello.ia` | 继续作为最小 smoke |
| 控制流示例 | `ialang/examples/control.ia` 或新增更小替代文件 | 形成稳定输出 |
| 函数示例 | `ialang/examples/function.ia` 或新增更小替代文件 | 覆盖多函数调用 |
| capability 示例 | 新增专门 IAVM smoke 示例 | 明确 profile / capability 行为 |

**任务:**
- 为每个纳入矩阵的示例定义：
  - 输入文件
  - 输出模块路径
  - 推荐 profile
  - 期望 stdout
- 对不稳定示例建立 issue/限制说明，不再模糊处理。

**验收标准:**
- 至少 3 个示例形成稳定 smoke matrix。
- 每个示例都有自动化测试或可复用命令。

### Milestone 4：旧示例兼容性修复

**目标:** 处理 `0.0.1` 中暴露出的“现有语言示例无法稳定进入 IAVM”问题。

**优先项:**
- `ialang/examples/hello.ia`

**任务:**
- 复现并定位 `cannot get property from non-object` 的根因。
- 判断问题属于：
  - lowering 错误
  - runtime 对象语义缺失
  - 内建 import / property access 兼容性问题
- 至少完成以下之一：
  - 真正修复，并加入回归测试；
  - 保持失败，但形成明确 regression test + documented limitation。

**验收标准:**
- 该问题从“已知但未定性”变为“已修复”或“已明确定性并被测试/文档覆盖”。

### Milestone 5：发布前测试矩阵

**目标:** 在 `0.0.1` 测试基线之上，增加 0.0.2 的平台硬化测试。

**必跑命令:**

```bash
go test ./iacommon/...
go test ./iavm/...
go test ./ialang/...
```

**必跑 smoke:**

```bash
go build -o ./bin/ialang ./ialang/cmd/ialang
./bin/ialang --help
./bin/ialang build-iavm ./ialang/examples/iavm_hello.ia -o ./tmp/iavm_hello.iavm
./bin/ialang verify-iavm ./tmp/iavm_hello.iavm --profile sandbox
./bin/ialang inspect-iavm ./tmp/iavm_hello.iavm --verify --profile sandbox
./bin/ialang run-iavm ./tmp/iavm_hello.iavm --profile sandbox
```

**新增 smoke 目标:**
- 至少 2 个额外示例进入自动化 smoke。
- profile/capability 组合行为有专门测试覆盖。

**验收标准:**
- 所有 Go 测试通过。
- 示例矩阵 smoke 通过。
- policy 输出格式和失败诊断格式稳定。

## 5. 风险与处理策略

| 风险 | 影响 | 处理策略 |
|---|---|---|
| policy 语义调整可能影响现有 CLI 测试 | 0.0.1 已有行为回归 | 先固化现有行为测试，再逐步替换为统一 policy 语义 |
| capability 语义收敛后可能暴露更多历史问题 | 更多示例转为失败 | 把失败转为显式、可诊断、可测试，而不是隐式通过 |
| 旧示例兼容问题根因可能涉及 lowering 和 runtime 两层 | 修复范围扩大 | 优先做问题定性与回归测试，必要时拆到 0.0.3 |
| 示例矩阵扩展过快 | 版本目标再次膨胀 | 只纳入 2~3 个真正稳定的 IAVM 示例 |

## 7. 0.0.2 之后的方向

当 `0.0.2` 完成后，再考虑进入更高一层的平台推进：

1. **更完整的 IAVM 示例与语言子集映射**
   - 系统梳理哪些 `ialang` 语义已经能稳定 lowering 到 IAVM。

2. **Host ABI 一致性校验**
   - capability 声明、导入、host call、runtime 执行路径统一约束。

3. **更正式的 sandbox 定义**
   - 从“调试/开发限制策略”推进到“更明确的安全边界定义”。

4. **0.0.3 候选方向**
   - 包/模块支持强化。
   - 更复杂示例兼容性修复。
   - 更系统的运行时语义补齐。
